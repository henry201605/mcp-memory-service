### 😤 痛点三：项目知识反复解释

**你的项目有很多「只有老人才知道」的潜规则**

我们舆情项目（CRSDP）的真实复杂度：

- **保存素材有 3 个独立实现**：Kafka 消费入库、舆情计划订阅入库、用户上传文件——三者共用 MaterialTable 和 FieldUtil，改一个要同步验证 **12 个文件**
- **预警邮件流程 6 步**：配置规则 → 每 10 分钟轮询（Redisson 分布式锁防并发）→ 每日去重 → ES 增量查询 → 发 HTML 邮件 → 写日志更新时间戳
- **站点数据走 gRPC**：从 UCF 超管系统获取，两层缓存（全量 30 天 + 院校 10 分钟），有效期分长期和试用两种判断逻辑
- **7 条核心调用链**：舆情入库、AI 对话、报告生成、素材管理、舆情搜索、认证拦截、gRPC 调用

**每次换个 AI 工具或开新对话，这些全要重新说一遍**

👉 **这就像公司没有文档，所有知识都在老员工脑子里**

---

### 😤 痛点四：跨项目经验无法复用

**在 A 项目踩的坑，到 B 项目又踩一遍**

真实案例——K8s 部署踩坑：

在部署 Memory Service 时踩了 6 个坑：
1. MCP_SSE_HOST 必须设为 0.0.0.0（默认 127.0.0.1 导致探针全部失败）
2. 首次启动需下载 90MB embedding 模型（Dockerfile 中要预装）
3. containerd 运行时无 /.dockerenv 文件（需要 monkey-patch）
4. SQLite 单写，replicas 只能设 1
5. 用 TCP Socket 探针而非 HTTP（服务没有 /health 端点）
6. startupProbe 要设 60 次重试（模型加载最多等 10 分钟）

**这些经验如果不记录，下个项目部署时又要重新踩一遍**

## 第四章：它是怎么工作的

---

### 🏗️ 整体架构（通俗版）

把它想象成一个「AI 的云笔记」：

```
你的 AI 工具（Kiro / Claude / Cursor）
        ↕ 通过 MCP 协议通信（就像 USB 接口）
   Memory Service（记忆服务）
        ↕
   数据库 + Embedding 模型
```

**就像你用印象笔记**：
- 写笔记 = AI 存储一条记忆
- 搜笔记 = AI 检索相关记忆
- 分类标签 = 按项目、按类型整理
- 多设备同步 = 多个 AI 工具共享同一套记忆

---

### 🔍 语义搜索 vs 关键词搜索

#### 传统搜索（关键词匹配）

搜「数据库连接超时」→ 只能找到包含这几个字的内容

#### 语义搜索（Memory Service 的方式）

搜「数据库连接超时」→ 还能找到：
- 「MySQL 连接池耗尽的排查过程」
- 「Redis 超时配置从 3s 改为 10s」
- 「生产环境 DB 响应变慢的根因分析」

#### 原理（简单版）

把每条记忆变成一组数字（向量），意思相近的记忆，数字也相近。搜索时比较数字的「距离」，而不是比较文字是否一样。

**就像图书馆：关键词搜索是按书名找书，语义搜索是按内容找书**

---

### 🏷️ 标签系统——项目隔离

```
project:crsdp         → 舆情项目：预警流程、素材管理、站点迁移……
project:unipus-agent   → 教育 Agent：作业发布、意图路由、工具调用……
project:ucloud         → 云作业项目的记忆
project:mcp-memory     → Memory Service 自身的架构和部署经验
```

**AI 处理舆情项目时，只检索 crsdp 的记忆，不会把作业项目的知识混进来**

---

### 📊 记忆的生命周期——模仿人脑

```
产生 → 存储 → 检索 → 整合 → 遗忘
```

**最有趣的是「整合」——模仿人类睡眠时的记忆整合**：

| 阶段 | 做什么 | 类比 |
|------|--------|------|
| 聚类 | 把相似记忆归到一起 | 整理书架，同类放一起 |
| 压缩 | 合并重复信息 | 3 篇笔记说同一件事，合成 1 篇 |
| 衰减 | 久不使用的记忆权重降低 | 半年没翻的笔记排到后面 |
| 遗忘 | 清理过时或低价值的记忆 | 扔掉过期的便签 |
| 关联 | 发现记忆之间的隐含联系 | 突然想到两件事有关系 |

**结果：记忆库始终保持「新鲜」和「精炼」**

---

### 🔌 MCP 协议——AI 工具的「USB 接口」

以前每个 AI 工具都有自己的插件格式，互不兼容。

MCP（Model Context Protocol）就像 USB 标准：
- 定义了统一的「插头」格式
- 任何 AI 工具都能用
- 插上就能用，不用改代码

**支持的 AI 工具**：

| 工具 | 说明 |
|------|------|
| **Kiro** | 我们正在用的 |
| Claude Desktop | Anthropic 官方客户端 |
| Claude Code | 命令行版 Claude |
| Cursor | AI 代码编辑器 |
| Windsurf | Codeium 的 AI 编辑器 |
| VS Code + Copilot | 微软的方案 |
| 还有 8+ 个 | 生态在快速扩展 |

**关键优势：今天用 Kiro 存的记忆，明天换 Cursor 也能用**

## 第五章：我们的部署方案

---

### 🚀 部署架构

```
开发者的 IDE（Kiro / Claude / Cursor）
        ↕ HTTPS
   K8s 集群中的 Memory Service
   （memory-mcp.unipus.cn/mcp）
        ↕
   SQLite 数据库（CFS 持久化存储）
   + ONNX 本地 Embedding 模型（~90MB）
```

---

### 🛡️ 为什么选择自托管？

| 对比项 | 云服务方案（如 Mem0 Cloud） | 我们的方案 |
|--------|--------------------------|----------|
| 数据安全 | 代码和业务数据上传到第三方 | **数据在自己的 K8s 集群** |
| 费用 | 按 API 调用收费 | **零额外费用** |
| 延迟 | 跨网络调用 | **内网访问** |
| 可控性 | 受限于服务商 | **完全自主** |
| 依赖 | 需要外部 LLM API | **本地 ONNX 模型，无外部依赖** |

**部署成本**：一个 Pod，约 800MB 内存，无需 GPU，无需额外数据库

---

### 🆚 跟竞品的对比

| 方案 | Stars | 优势 | 劣势 |
|------|-------|------|------|
| **MCP Memory Service** | — | K8s 自托管、多工具共享、零 API 费用 | 需要初始配置 |
| claude-mem | 46K | 全自动 Hook 捕获 | 锁定 Claude 生态，有 API 费用 |
| Mem0 | 47K | 图谱能力强 | 部署重（需向量库+图库+LLM API） |
| Basic Memory | — | Markdown 存储人类可读 | 只适合 Obsidian 用户 |
| Claude 内置记忆 | — | 零配置 | 不能跨工具，不能团队共享 |

## 🙋 Q&A

### 常见问题

**Q：会不会记错东西？**
A：有质量评估系统（AI 评分 + ONNX 排序模型 + 隐式信号），低质量记忆自动降权。也可以手动删除错误记忆。

**Q：记忆会不会越来越多变慢？**
A：有自动整合机制（聚类、压缩、衰减、遗忘），就像人脑会整理和遗忘不重要的信息。10 万条记忆检索 < 100ms。

**Q：不同项目的记忆会混吗？**
A：通过标签隔离。每个项目有独立标签前缀（如 project:crsdp），搜索时自动过滤。

**Q：我需要做什么？**
A：在你的 AI 工具里配置 MCP 连接地址就行，5 分钟搞定。之后 AI 会自动存储和检索记忆。

**Q：支持哪些 AI 工具？**
A：14+ 个 AI 客户端，包括 Kiro、Claude Desktop、Claude Code、Cursor、VS Code Copilot、Windsurf 等。

**Q：数据安全吗？**
A：所有数据在我们自己的 K8s 集群，Embedding 模型本地运行（ONNX），不调用任何外部 API，不上传任何代码到第三方。
